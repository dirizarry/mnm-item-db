# global-metadata.dat encryption — analysis notes

Durable findings from the metadata decryption effort (this file is **not** regenerated by the
pipeline, unlike `CLIENT-RE.md`).

## Summary

| Property | Value |
|----------|-------|
| On-disk header magic | `9805678d` (standard IL2CPP magic is `0xFAB11BAF` / on-disk `af1bb1fa`) |
| Metadata wire version | 39 (Unity 6000.0.59f2) |
| File size | 20,401,936 bytes |
| Encrypted region | First **380 bytes** (the v39 header: magic + version + 31 × `{offset,size,count}`) |
| Body | **Plaintext** — string heaps & tables intact (`ItemRecord`, `System.`, `mscorlib`, …) |
| Runtime (`mnm.exe`) | ~20.4 MB mirror has the **same** encrypted header; no decrypted copy in RAM |

The body being plaintext but the header encrypted means the loader very likely decrypts the
header (or individual fields) on access rather than decrypting the whole file in place — which is
why no `0xFAB11BAF` buffer ever appears in process memory.

## Methods attempted (all failed to produce a valid header)

1. Flat repeating XOR, key lengths 4/8/16/24/32 (magic+version derived) — magic restores, offsets garbage.
2. disunity-style Kasiski + known-plaintext (partial-from-byte-7 and full-from-byte-0) — no valid key.
3. Magic-only patch / self-referential keys — no valid header.
4. Linear positional keystream (xor/add/sub, slopes ±1..±3, all intercepts) — see lead below.
5. Per-field single-byte XOR brute on each header dword — no consistent key.
6. ASCII-key brute inside `GameAssembly.dll` (decrypt magic+v35..41) — 0 hits.
7. Process-memory scan for `0xFAB11BAF` / valid v39 header — 0 hits (game in-game, character loaded).

## Lead for interactive RE (Ghidra/IDA)

Sampling the encrypted header at every 4th byte — the offset/size/count **MSB** positions, where
the true plaintext is ~0 because most table offsets are < 16 MB — reveals a clean keystream:

```
S[p] ≈ (144 - p) & 0xFF      # slope -1 per byte, ~93 consecutive consistent samples
```

±1 jitter appears exactly where the true offset ≥ 16 MB (MSB = `0x01`).

BUT this ramp only fits the MSB bytes. Decrypting the whole header with it yields a wrong magic
(`0x00e98a08`) and non-monotonic offsets, and bytes 0–7 use a different transform (true XOR key
there is `37 1e d6 77 97 cc bc 88`). So the keystream is a non-positional sequence whose MSB bytes
*happen* to ramp. Recovering the rest requires reading the actual decrypt routine.

### Where to look in `GameAssembly.dll`

- Exports are obfuscated (random 11-char names); no `il2cpp_*` exports present.
- Trace `MetadataLoader::LoadMetadataFile` → `MetadataCache::Initialize` (the `sanity == 0xFAB11BAF`
  assert site). The decrypt happens between the file read and the header dereference.
  See [Katyscode tutorial](https://katyscode.wordpress.com/2021/02/23/il2cpp-finding-obfuscated-global-metadata/).
- Alternatively hook the header read at runtime with Frida
  ([Il2CppMetadataExtractor](https://github.com/CameroonD/Il2CppMetadataExtractor)); note the
  per-field decrypt theory means you may need to hook the field accessor, not a one-shot buffer.

# Where the item/monster data actually lives

Investigated the Addressable bundles (the "Odin track") to see if item/monster **stats** ship
on disk. They do **not**:

| Bundle | Contents (asset-type census) |
|--------|------------------------------|
| `defaultlocalgroup` (139 MB) | 855 Sprites (UI icons), 254 Texture2D, Shaders, VFX, HDRP volume MonoBehaviours. Largest asset = a 707 KB Shader. **No data tables.** |
| `contentupdate` (22 MB) | 419 AnimationClips + skinned meshes (character animations). 2 MonoBehaviours. |
| `globalitems` (7.4 MB) | 5,839 GameObjects + 522 Meshes + 1,083 MonoBehaviours = **equipment 3D models** (e.g. `Arabian_Axe1H_00`, `WeatheredLeatherB_Helm_HumanF_Base`). |

Conclusion: M&M is server-authoritative. Item/monster **stats** are delivered at runtime — which
is exactly what the **ledger** pipeline captures. The bundles only carry **presentation** assets.

## What we extract from the client (`client_re/item_assets.py`)

- **603 icon sprites** with internal-key-style names (`boots_steel_equipment`, `apple_food`,
  `backpack_leather_container`) — convention is roughly `{name}_{material}_{category}`.
- **246 ability/class UI sprites** (separated out).
- **814 equipment model names** from `globalitems` (rig/bone nodes filtered).
- **Fuzzy wiki crosswalk** (token-based, many-to-many): 122 icons matched to wiki titles at
  score ≥ 0.5 (exact hits like `Arrow`, `Bell Pepper`, `Cabbage`, `Campfire`).

Output: `data/client/item-assets.json`. Optional icon PNG export:

```bash
python mnm_client_db.py --item-assets                # catalog + crosswalk
python mnm_client_db.py --item-assets --export-icons  # also dump PNGs to data/client/icons/
```
